第五章-创建高性能的索引

发表于 2020-09-03 更新于 2021-02-19

索引基础

索引的类型

索引在存储引擎层实现

B-Tree索引

单列索引:

多列索引:

索引查询原则:

  • 最左前缀匹配, 从索引定义的最左列开始, 可使用全部或部分列的索引
  • 如果某个列有范围查询, 则其右边的列都无法使用索引

哈希索引

只有Memory引擎支持哈希索引, 也是Memory的默认索引类型.

特性

  • 只包含索引值和行指针, 不能用索引值避免读取行
  • 无法用于排序
  • 哈希索引对所有索引列进行hash, 必须使用部分列查索引
  • 只支持=、IN()、<=>, 不支持范围查询
  • 冲突小时查询很快, 冲突严重时查询和删除很慢

InnoDB在某些索引值操作频繁时会自发的在B-Tree索引上再创建一个哈希索引 (只是将key进行hash缩短键值, 还是使用B-Tree查找)

不支持哈希索引的存储引擎伪hash索引的实现

如 (mysql会选择url_crc列): 为什么将长键转为短键能缩短搜索空间???

对于url_crc列可定义触发器进行维护

WHERE条件需要包含原来的列, 用于处理哈希冲突

空间数据索引 (R-Tree)

用于地理数据索引, 不需要前缀查询, 可以利用任意维度来查询

MyISAM支持

PostgreSQL和PostGIS数据库做的较好

全文索引

对全文的关键词匹配, 和基于值匹配的B-Tree索引不冲突

适用于MATCH AGAINST操作

索引的优点

三个优点:

  • 减少扫描的数据量
  • 避免排序和临时表
  • 随机IO变为顺序IO

tips:

  • 小表全盘扫更高效
  • 中到大型的表用索引很有效
  • 特大表见索引代价高, 考虑分区
  • 表的数量太多, 考虑元数据信息表

高性能的索引策略

独立的列

索引列不能是表达式的一部分, 也不能是函数的参数

前缀索引和索引选择性

完整列的选择性: SELECT COUNT(DISTINCT city)/COUNT(*) FROM sakila.city_demo;

通常前缀索引的长度应尽量接近完整列的选择性

此外还需要考虑前缀索引每个类型的区分性

前缀索引无法做ORDER BY和GROUP BY

不需要修改表结构

后缀索引可以将字符串反转存储后建立前缀索引 (借助触发器自动触发)

前缀索引创建: ALTER TABLE sakila.city_demo ADD KEY (city(7));

多列索引

多个单列的索引在查询条件中使用时, MySQL可能会对OR和AND条件使用UNION和INTERSECTION优化

如下使用了actor_id和film_id两个单列索引, 使用了两个索引扫描的联合:

EXPLAIN SELECT film_id, actor_id FROM sakila.film_actor WHERE actor_id=1 OR film_id=1 \G

Tips:

  • INTERSECTION的存在通常意味着需要联合索引
  • OR操作往往会合并返回大量数据, 占用大量CPU和内存资源
  • 优化器更在意随机页面读取, 通常会低估查询成本. 使用多列索引消耗更多CPU和内存资源且可能影响并发性. 通常不如直接改写成UNION方式更好
  • optimizer_switch可便宜关闭索引合并功能
  • IGNORE INDEX提示优化器忽略掉某些索引

选择合适的索引列顺序

不考虑排序和分组时, 通常考虑将选择性最高的列放在前面

通过多个列的选择性考虑联合索引的顺序 (经验法则):

聚簇索引

数据行和键值紧凑的存储在一起, 一个表只能有一个聚簇索引

InnoDB的聚簇索引:

  • 默认主键为聚簇索引
  • 如果没有主键, 会选择一个唯一的非空索引
  • 如果没有这样的索引, 会隐式定义一个逐渐作为聚簇索引

InnoDB只聚集同一个页面的记录, 包含相邻键值的页面可能相聚甚远

优点:

  • 相关的数据存储在一起. 如根据用户ID为聚簇索引, 该用户的所有邮件只用很少磁盘IO
  • 数据和索引一起存储, 查找快
  • 使用覆盖索引扫描的查询可以直接使用页节点中的主键值 ???

缺点:

  • 数据量少能放在内存中时, 聚簇索引没什么优势
  • 如果不是按索引顺序插入数据, 可以用OPTIMIZE TABLE重新组织一下表
  • 更新聚簇索引代价高, 被更新的行会被移动到新的位置
  • 插入新行或更新主键会导致“页分裂”问题, 占用更多磁盘
  • 行稀疏或数据存储不连续时, 全表扫描慢
  • 二级索引会存储主键列, 且需要两次查找 (至少)

InnoDB自适应的哈希索引能减少二级索引的多次查找???

InnoDB和MyISAM的数据分布对比

InnoDB主键为聚簇索引, 其他索引为存储主键的二级索引

MyISAM中主键索引和二级索引都存储数据的行号

在InnoDB表中按主键顺序插入行

向聚簇索引顺序插入行:

  • 新插入的row被顺序写入到磁盘
  • 高并发情况下造成间隙锁竞争
  • 高并发情况下AUTO_INCREMENT锁的影响, 可以调节innodb_autoinc_lock_mode参数改善(ref: https://www.cnblogs.com/JiangLe/p/6362770.html)

向基簇索引插入乱序的行:

  • 目标页可能不在磁盘中, 加载会造成许多随机IO
  • 乱序写入有频繁页分裂, 分配新的空间, 一次插入最少需要修改三个页
  • 页分裂使得页变得稀疏, 存在碎片

覆盖索引

MySQL利用索引列直接获取列数据, 这样就不再需要回表查询读取数据行.

EXPLAIN的EXTRA: using index为使用了覆盖索引

优势:

  • 索引比数据行小, 只读取索引能大大减少数据访问量
  • 索引按顺序存储 (至少单页内), 随机IO比读取数据行少
  • 一些存储引擎如MyISAM只缓存索引, 数据则依赖操作系统缓存, 每次访问数据需要一次系统调用, 开销大
  • InnoDB二级索引中保存了主键值, 可以覆盖对于主键的查询

MySQL能在索引中做最左前缀的LIKE比较, 但对于%开头的比较就无法利用索引

使用索引扫描做排序

使用索引排序的情况:

  • 索引的列顺序和ORDER BY子句的顺序完全一致, 并且所有列的排序方向都一样时
  • ORDER BY子句要求和WHERE的要求一样, 满足索引最左前缀要求. 前导列在WHERE或JOIN子句中指定了常量时, 可以不满足最左前缀要求
  • 需要多表关联时, 只有当ORDER BY子句引用的字段都为第一个表时, 才能用索引做排序. (依赖优化器使用哪张表作为第一张表)

EXPLAIN的type: index为使用索引做排序

tips:

  • 如果不同列需要按不同方向做排序, 可以将存储该列值进行反转串或相反数

压缩(前缀压缩)索引

MyISAM使用前缀压缩来减少索引大小, 让更多的索引可以放入内存. 默认值只压缩字符串, 通过参数设置页可以对整数做压缩

MyISAM对行指针也采用类似的前缀压缩方式

索引压缩的方法:

  • 先完全保存索引块中第一个值, 然后将其他值和第一个值进行比较得到相同前缀的字节数和剩余的不同后缀部分
  • 如: 索引快第一个值为“perform”, 第二个值是“performace”, 第二个值采用“7,ance”这样的形式

优劣:

  • IO密集型应用, 对有些查询带来的好处比成本多很多
  • 因为后一个索引值依赖前一个值, 所以ORDER BY DESC性能不好
  • CPU密集型应用, 因为需要随机查找, 压缩索引在索引块内的查找是顺序扫描不是二分, 索引性能慢

冗余和重复索引

尽量避免冗余和重复索引, 但由于联合索引占空间大, 所以有时为了加速不同query查询会存在冗余索引

为什么前缀压缩的MyISAM使用索引(state_id, city, address)的情况下该语句查询效率比单列索引下降很多呢, 而InnoDB不会???

分析表中冗余索引和重复索引的方法:

  • 复杂的访问INFORMATION_SCHEMA表的查询
  • Sholomi Noach的common_schema中的一些视图定位, commom_schema是一系列安装到服务器上常用的存储和试图
  • Percona Toolkit中的pt-duplicate-key-checker工具来检查

未使用的索引

不使用索引的查询方式:

  • Percona Server或MariaDB先打开userstates服务器变量 (默认关闭), 查询INFORMATION_SCHEMA.INDEX_STATISTICS查询每个索引的使用频率
  • Percona Toolkit中的pt-index-usage分析未使用的索引, 并定位质量差的查询, 结果可卸乳MySQL的表中

索引和锁

查询会先通过存储引擎索引过滤返回, 对于索引无法过滤的数据, 存储引擎返回后通过MySQL服务器WHERE子句过滤

如果不使用索引, InnoDB会做全表扫描锁住所有的行

如下例子InnoDB锁定14行, 但实际上只返回了24

InnoDB二级索引使用共享锁, 主键索引使用排他锁. 所以获取写锁的时候无法使用覆盖索引

第四章 Schema与数据类型优化

发表于 2020-08-31 更新于 2020-09-03

几个原则

  • 选择更小且够用的数据类型

  • 用简单的数据类型

    如使用整型存IP, 用内置时间类型存时间而不用字符类型

  • 尽量避免NULL

    NULL值更难优化, 使用更多存储空间, 索引需要多记录信息. 例外: InnoDB使用位存储NULL, 更是用于稀疏数据存储

基本数据类型

整数类型

TINYINT (8)、SMALLINT (16)、MEDIUMINT (24)、INT (32)、BIGINT (64), 括号中为存储占用的位空间

定义字段时可以指定宽度, 如INT(8), 这里8只是显示宽度, 并不影响内部实际的存储空间占用

IPv4最好用32位整数存储, 借助INET_ATON()和INET_NTOA()方便转换

实数类型

FLOAT、DOUBLE分别为4字节和8字节

DECIMAL支持精确计算, 数字保存为二进制字符串 (每4个字节存9个数字), 允许最多65个数组. DECIMAL(18, 9) 小数点前后各存储9个数字, 占用9个字节 (整数4+小数4+小数点1).

指定小数点后精度情况下可以用BIGINT代替DECIMAL避免高昂开销

字符串类型

  • VARCHAR

    1/2字节空间记录长度 (1: 长度<=255, 否则2), 保留末尾空格.VARHCHAR(10) 11字节, VARCHAR(1000) 1002字节. update记录变长时, 可能分段或分裂页存储到页内, 不同存储引擎不一样

  • CHAR

    定长, 截断末尾空格. 截断在存储引擎层处理

  • BINARY、VARBINARY

    存储二进制字符串, 使用\0填充, 检索时也不会去掉填充值. 按字节比较, 速度快

  • BLOB和TEXT

    TINYTEXT、SMALLTEXT、TEXT、MEDIUMTEXT、LONGTEXT: 有字符集和排序规则

    TINYBLOB、SMALLBLOB、BLOB、MEDIUMBLOB、LONGBLOB: 二进制存储, 没有字符集和排序规则

    不对字段的整个内容做排序: 调整max_sort_length或ORDER BY SUSTRING(column, length)

    MySQL不对全部长度的字符串进行索引, 也不能用这些索引消除排序

    内存临时表大小限制: max_heap_table_size或tmp_table_size, 超过后转为使用磁盘临时表

  • ENUM

    可用于代替字符串, 实际存储的是整数

    增加字符串枚举类型定义需要ALTER TABLE, 开销大

    ORDER BY默认以内部存储的整数排序, ORDER BY FILED(e, ‘apple’, ‘dog’, ‘fish’)按字符串排序

    两个表的关联混用ENUM和VARCHAR速度会很慢

日期和时间类型

精确到秒

  • DATETIME

    8字节存储, 日期和时间YYYYMMDDHHMMSS封装到整数中, 与时区无关

    范围1001年~9999年

  • TIMESTAMP

    4字节存储, 时间戳

    范围1970年~2038年

    MySQL在插入和更新时, 如果没有指定第一个TIMESTAMP列的值, 会更新第一个TIMESTAMP列的值

    默认为NOT NULL

  • BIGINT/DOUBLE: 可用于存储更高精度的时间, ms

位数据类型

  • BIT: 多个BIT列是否被整合存储视存储引擎而定. MyISAM整合存储, Memory和InnoDB并不是

    char(0)可以保存空值NULL和空字符串’’, 也可以达到BIT(1)的作用

  • SET: 用于保存多个true/false值, 例如ACL访问控制列表. 显示和比较时以字符串形式

    FIND_IN_SET(‘CAN_READ’, perms)

选择标识符 (Identifier)

标识符列可能用于外键, 通过标识符查找其他列

标识符的列最好用递增的整数类型. ENUM、SET、字符串类型、MD5随机值等都不是好的选择

标识符列应该和关联表中列的类型一致

不当的Schema设计

  • 太多的列
  • 太多的关联, 一次查询的关联最好在12个表以内
  • 滥用枚举和SET等, 如enum(‘’, ‘0’, ‘1’, ‘2’, …, ‘31’), 应该增加外键. SET(‘Y’, ‘N’)应该用enum
  • 尽力避免使用NULL, 但不要让数据库充满无意义的值

范式和反范式

  • 范式的优缺点

    更新操作快、较少或没有重复数据、表更小可以放到内存、检索时很少需要DISTINCT或者GROUP BY

    稍微复杂的查询语句可能就需要很多关联查找

  • 反范式的优缺点

    避免关联查找、全表扫描时避免随机IO

    更新时需要更新多张表

总的来说, 生产环境下会混合使用范式和反范式, 比如用于复制或者缓存

缓存表和汇总表

  • 缓存表: 可以简单的从其他表获取数据的表, 但是每次获取的速度慢
  • 汇总表: 保存使用了GROUP BY等语句聚合数据的表

物化视图

物化视图是预先计算并存储在磁盘上的表, 可以通过各种各样的策略刷新和更新

MySQL不原生的支持物化视图, 借助开源工具Flexviews. Flexviews通过提取源表的更改, 基于行的二进制日志, 不需要使用查询语句

计数器表

计数器可以记录在表中, 但是这样会存在并发问题. 计数器可以单独存表, 这样能利用一些更高级的技巧.

  • 按天生成计数器
  • 利用slot提升更新并发效率
  • 可以周期性执行任务合并每天的结果到slot 0, 并删除其他的槽

更快的读, 更慢的写

ALTER TABLE

常见的加速方法:

  • 在一台不提供服务的机器上执行ALTER TABLE, 然后与主库切换
  • 影子拷贝: 复制源表, 在复制表上操作, 然后重命名交换源表

修改Column默认值例子:

  • 慢, 读取全表并插入

  • 快, 只修改.frm文件 (存储表的元数据), 不需要重建表

只修改.frm文件

不需要重建表的操作:

  • 移除一个列的AUTO_INCREMENT属性
  • 增加、移除或更改ENUM和SET常量. 被移除的行被查询后会返回一个空字符串

技巧:

  1. 创建有相同结构的空表, 进行修改. CREATE TABLE … LIKE …. ALTER …
  2. 执行FLUSH TABLES WITH READ LOCK, 关闭使用的表, 并禁止任何表打开
  3. 交换.fim文件. mv new.frm old.frm
  4. 执行UNLOCK TABLES

快速创建MyISAM索引

载入全部数据后构建索引会快很多, 可以利用排序构建索引, 且索引树的碎片更少.

方法一:

  1. DISABLE KEYS禁用索引
  2. load the data
  3. ENABLE KEYS

方法二: 影子文件

  1. 创建相同结构的表单不包含索引
  2. 载入数据到表中生成.MYD文件
  3. 创建另一张包含索引的表的结构, .firm和.MYI
  4. 获取读锁因刷新表
  5. 将第二张表重命名为第一张表
  6. 释放读锁
  7. 使用REPAIR TABLE来重建表的索引, 会通过排序创建, 包括唯一索引.
